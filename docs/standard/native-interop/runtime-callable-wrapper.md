# <a name="runtime-callable-wrapper"></a><span data-ttu-id="6da2e-103">ランタイム呼び出し可能ラッパー</span><span class="sxs-lookup"><span data-stu-id="6da2e-103">Runtime Callable Wrapper</span></span>
<span data-ttu-id="6da2e-104">共通言語ランタイムは、ランタイム呼び出し可能ラッパー (RCW) と呼ばれるプロキシを介して COM オブジェクトを公開します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-104">The common language runtime exposes COM objects through a proxy called the runtime callable wrapper (RCW).</span></span> <span data-ttu-id="6da2e-105">.NET クライアントでは RCW が普通のオブジェクトとして示されますが、その主な機能は、.NET クライアントと COM オブジェクトの間で呼び出しをマーシャリングすることです。</span><span class="sxs-lookup"><span data-stu-id="6da2e-105">Although the RCW appears to be an ordinary object to .NET clients, its primary function is to marshal calls between a .NET client and a COM object.</span></span>  
  
 <span data-ttu-id="6da2e-106">ランタイムは、COM オブジェクトごとに 1 つの RCW を、そのオブジェクトに存在する参照の数に関係なく作成します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-106">The runtime creates exactly one RCW for each COM object, regardless of the number of references that exist on that object.</span></span> <span data-ttu-id="6da2e-107">ランタイムは、各オブジェクトで、プロセスごとに 1 つの RCW を保持します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-107">The runtime maintains a single RCW per process for each object.</span></span>  <span data-ttu-id="6da2e-108">1 つのアプリケーション ドメインまたはアパートメントで RCW を作成し、別のアプリケーション ドメインまたはアパートメントへの参照を渡すと、最初のオブジェクトへのプロキシが使用されます。</span><span class="sxs-lookup"><span data-stu-id="6da2e-108">If you create an RCW in one application domain or apartment, and then pass a reference to another application domain or apartment, a proxy to the first object will be used.</span></span>  <span data-ttu-id="6da2e-109">次の図が示すように、任意の数のマネージド クライアントが、INew および INewer のインターフェイスを公開する COM オブジェクトへの参照を保持できます。</span><span class="sxs-lookup"><span data-stu-id="6da2e-109">As the following illustration shows, any number of managed clients can hold a reference to the COM objects that expose INew and INewer interfaces.</span></span>  

<span data-ttu-id="6da2e-110">次の図は、ランタイム呼び出し可能ラッパー経由で COM オブジェクトにアクセスするプロセスを示しています。</span><span class="sxs-lookup"><span data-stu-id="6da2e-110">The following image shows the process for accessing COM objects through the runtime callable wrapper:</span></span>

 ![RCW 経由で COM オブジェクトにアクセスするプロセス。](./media/runtime-callable-wrapper/runtime-callable-wrapper.gif)  

 <span data-ttu-id="6da2e-112">ランタイムは、タイプ ライブラリから派生したメタデータを使用して、呼び出される COM オブジェクトとそのオブジェクトのラッパーの両方を作成します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-112">Using metadata derived from a type library, the runtime creates both the COM object being called and a wrapper for that object.</span></span> <span data-ttu-id="6da2e-113">各 RCW は、ラップする COM オブジェクトでインターフェイス ポインターのキャッシュを保持し、RCW が不要になった時点で、COM オブジェクトでのその参照を解放します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-113">Each RCW maintains a cache of interface pointers on the COM object it wraps and releases its reference on the COM object when the RCW is no longer needed.</span></span> <span data-ttu-id="6da2e-114">ランタイムは、RCW でのガベージ コレクションを実行します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-114">The runtime performs garbage collection on the RCW.</span></span>  
  
 <span data-ttu-id="6da2e-115">RCW は、他のさまざまなアクティビティも実行しますが、ラップされたオブジェクトのために、マネージド コードとアンマネージド コードの間でデータをマーシャリングします。</span><span class="sxs-lookup"><span data-stu-id="6da2e-115">Among other activities, the RCW marshals data between managed and unmanaged code, on behalf of the wrapped object.</span></span> <span data-ttu-id="6da2e-116">具体的には、RCW はクライアントとサーバーの間でデータの異なる表現が渡されるとき、常にメソッド引数やメソッドの戻り値のマーシャリングを提供します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-116">Specifically, the RCW provides marshaling for method arguments and method return values whenever the client and server have different representations of the data passed between them.</span></span>  
  
 <span data-ttu-id="6da2e-117">標準ラッパーは、組み込みのマーシャリング規則を適用します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-117">The standard wrapper enforces built-in marshaling rules.</span></span> <span data-ttu-id="6da2e-118">たとえば、.NET クライアントが文字列型を引数の一部としてアンマネージ オブジェクトに渡すとき、ラッパーは文字列を BSTR 型に変換します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-118">For example, when a .NET client passes a String type as part of an argument to an unmanaged object, the wrapper converts the string to a BSTR type.</span></span> <span data-ttu-id="6da2e-119">COM オブジェクトが、マネージド呼び出し元に BSTR を返す場合、呼び出し元は文字列を受信します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-119">Should the COM object return a BSTR to its managed caller, the caller receives a String.</span></span> <span data-ttu-id="6da2e-120">クライアントとサーバーはどちらも、それぞれが使い慣れているデータを送受信します。</span><span class="sxs-lookup"><span data-stu-id="6da2e-120">Both the client and the server send and receive data that is familiar to them.</span></span> <span data-ttu-id="6da2e-121">その他の型は、変換する必要がありません。</span><span class="sxs-lookup"><span data-stu-id="6da2e-121">Other types require no conversion.</span></span> <span data-ttu-id="6da2e-122">たとえば、標準的なラッパーはマネージド コードとアンマネージド コードの間で、型を変換しないで 4 バイトの整数を常に受け渡しします。</span><span class="sxs-lookup"><span data-stu-id="6da2e-122">For instance, a standard wrapper will always pass a 4-byte integer between managed and unmanaged code without converting the type.</span></span>
